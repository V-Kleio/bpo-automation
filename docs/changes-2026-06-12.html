Bug · High

Fix 1 — Chat stream leaves UI permanently locked on error

The for await (const chunk of streamAskAI(…)) loop in AskAIChat.tsx had no error handling. If the stream errored mid-message (network drop, server error, Claude API timeout), the assistant message stayed streaming: true forever and the input stayed disabled — the UI was completely locked until a page reload.

Fix: The loop is now wrapped in try/catch/finally. On error, the partial response is preserved in the message and an error notice is appended so the user can see what failed. streaming: false and setBusy(false) are guaranteed to run via finally, regardless of outcome.

Bug · High

Fix 2 — Queue item stuck in "sending" if worker crashes unexpectedly

sendOne() always finishes an item, but runWorker() had no top-level guard. If selectAdapter() or the inter-send sleep threw, the worker exited with an item still at status: "sending". Since workerPromise was then set to null, no future enqueue would recover that item — it was stuck forever.

Fix: The .finally() block in startWorker() now scans all items and resets any still at "sending" to "failed" with a descriptive recovery message.

Bug · Medium

Fix 3 — StepActions direct-send note truncated at 280 chars instead of 200

The "direct send" path in StepActions.tsx used body.slice(0, 280) for the connection note, while LinkedIn's free-account cap is 200 characters. Notes between 201–280 characters were sent over the limit and arrived chopped mid-word on the recipient's side.

Fix: Replaced body.slice(0, 280) with the shared truncateNote(body) helper from lib/services/linkedin/types.ts, which trims at a word boundary with an ellipsis — consistent with the queue panel path.

Bug · Medium

Fix 4 — Rate limiter file write not atomic (TOCTOU race)

writeUsage() wrote directly to usage.json with fs.writeFileSync. If the queue worker and the /api/linkedin/send direct endpoint both called acquireSlot() at the same moment, both could read count=N, both write count=N+1, and one increment would be silently lost — under-counting against the daily cap.

Fix: Adopted the same atomic write-to-temp + rename pattern used in lib/services/settings.ts. The rename is atomic on all supported platforms, so a concurrent reader always sees either the old or the new file — never a partial write.

New feature · High impact

★ New — Failed LinkedIn sends no longer consume the daily quota

Previously, the daily rate-limit slot was reserved before the send attempt. A failed send (wrong button click, LinkedIn rejection, Pending verification failure) still counted against the daily cap — penalising every retry and eating into the real allowance.

Fix: The slot is now only counted after a confirmed success. Both the queue worker and the direct-send route check getUsage().remaining before attempting the send (for an early-fail on cap reached), then call acquireSlot() only when result.success === true. A releaseSlot() function was also added to rate-limiter.ts for any future rollback needs.

Error handling · Medium

Fix 5 — Queue panel silently shows stale data when status endpoint fails

The refresh() polling callback had a bare catch { // ignore }. If the /api/linkedin/queue/status endpoint returned a 5xx or timed out, the queue panel continued displaying the last known counts with no indication they were stale.

Fix: Added statusStale state. When a poll fails (non-ok response or thrown error), a small amber badge appears in the panel heading: "status unavailable — retrying…". It clears automatically on the next successful refresh. The "Clear history" and "Cancel pending" buttons now also check res.ok and show a toast.error if the server rejects the request, rather than silently calling refresh() as if the clear succeeded.

Error handling · Medium

Fix 6 — Session expiry mid-batch burns rate-limit slots and time

If the LinkedIn session expired during a batch run, the first item failed with a "session redirected to login/checkpoint" error. The worker then continued processing the remaining "pending" items — all of which would fail the same way — wasting the inter-send delay and rate-limit checks on doomed requests.

Fix: After sendOne() completes, the worker inspects the error string. If it matches the session-expiry pattern (/login|checkpoint|session.*no longer valid|authwall/i), all remaining pending items are immediately failed with a clear message: "LinkedIn session expired — reauthenticate via the Connect LinkedIn button and re-queue." The worker then exits cleanly. A shared failRemaining() helper is also used by the existing mock-provider fast-fail path.

QoL · Medium

Fix 7 — HubSpot sync fetches only first 100 contacts (no pagination)

fetchHubSpotContacts() called getPage(100, …) exactly once, capping results at 100 contacts regardless of portal size. Portals with more contacts would silently miss the rest with no indication in the UI.

Fix: Implemented cursor-based pagination using HubSpot's page.paging.next.after cursor. The function now loops until there are no more pages or until a configurable hard cap of 500 contacts (5 pages × 100) is reached — preventing unbounded fetches on large portals. The unused limit parameter was removed from both the function signature and the API route.

New feature · Layer 1

◆ MySQL Database as Layer 1 Data Source

Layer 1 (Lead Database) can now pull leads directly from a local MySQL Docker container (bpo-mysql) in addition to CSV imports and HubSpot sync. A Sync MySQL button appears on the Leads page whenever the database is configured — it mirrors the same pattern as the HubSpot sync button.

Auto-discovery: The route queries information_schema and selects the first table named leads, contacts, companies, prospects, or targets; falls back to the first table in the database. Up to 500 rows are fetched and mapped to Company + Stakeholder records. Rows with the same company name are grouped; each row contributes at most one stakeholder.

Column mapping is case-insensitive and flexible — common names like company_name, company, contact_name, linkedin_url, email, title, industry, headcount, hq are recognised automatically. Unknown columns are ignored.

Settings page: All five connection parameters (DB_TYPE, DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) are now exposed in the MySQL Database group on the /settings page. The password field is masked as a secret. Saving new credentials resets the connection pool so the change takes effect immediately without a restart.

Default values (set in .env.local): host 127.0.0.1, port 3306, database bpo_db, user root, password mypassword.

Start the container with:
docker run --name bpo-mysql -e MYSQL_ROOT_PASSWORD=mypassword -e MYSQL_DATABASE=bpo_db -p 3306:3306 -d mysql:8
